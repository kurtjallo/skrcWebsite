This task ensures the global CSS and root layout have all the accessibility foundations in place. These are site-wide rules that apply to every page.

**1. Verify and enforce `<html lang="en-GB">` (A11Y-01):**
- Open `src/app/layout.tsx` and confirm the `<html>` element has `lang="en-GB"`. This was set in Phase 1 but verify it survived subsequent changes.
- If missing, add it. If set to "en" instead of "en-GB", correct it.

**2. Global focus-visible styles (A11Y-06):**
- Open `src/app/globals.css` and verify the global `*:focus-visible` rule exists with gold ring styling.
- The rule should be:
  ```css
  *:focus-visible {
    outline: 2px solid var(--color-accent-500);
    outline-offset: 2px;
  }
  ```
- Search the ENTIRE codebase for any instances of `outline: none` or `outline: 0` that are NOT paired with a replacement focus indicator. Remove any bare `outline: none` declarations.
- Search for any `focus:outline-none` Tailwind classes that are NOT accompanied by `focus-visible:ring-*` replacements. Fix these by adding `focus-visible:ring-2 focus-visible:ring-accent-500 focus-visible:ring-offset-2`.
- Check shadcn/ui components in `src/components/ui/` -- shadcn sometimes uses `focus-visible:outline-none` paired with ring styles, which is correct. Leave those as-is.

**3. Reduced motion media query (A11Y-09):**
- Verify `src/app/globals.css` contains the reduced motion media query:
  ```css
  @media (prefers-reduced-motion: reduce) {
    *, *::before, *::after {
      animation-duration: 0.01ms !important;
      animation-iteration-count: 1 !important;
      transition-duration: 0.01ms !important;
      scroll-behavior: auto !important;
    }
  }
  ```
- This was added in Phase 1 globals.css but verify it is still present.
- Search for any Motion (Framer Motion) `<motion.div>` components and ensure they respect reduced motion. Look for components using `whileInView`, `animate`, or `initial` props. These should either:
  a. Already use the `useReducedMotion` hook from Motion, OR
  b. Be wrapped in conditional logic that checks `prefers-reduced-motion`
- If any animated components (particularly in `src/components/shared/AnimatedSection.tsx` or similar) do NOT respect reduced motion via the hook, add the `useReducedMotion` hook:
  ```tsx
  import { useReducedMotion } from "motion/react";

  export function AnimatedSection({ children, ...props }) {
    const shouldReduceMotion = useReducedMotion();

    return (
      <motion.div
        initial={shouldReduceMotion ? false : { opacity: 0, y: 30 }}
        whileInView={shouldReduceMotion ? undefined : { opacity: 1, y: 0 }}
        // ... rest of props
      >
        {children}
      </motion.div>
    );
  }
  ```

**4. Touch target minimum sizes (A11Y-08):**
- Add a global CSS rule in `src/app/globals.css` that ensures minimum touch target sizes on all interactive elements:
  ```css
  /* Minimum touch target size (A11Y-08) */
  @media (pointer: coarse) {
    a, button, [role="button"], input, select, textarea, summary {
      min-height: 44px;
      min-width: 44px;
    }
  }
  ```
- Search for any buttons or links in components that might have explicit small heights (e.g., `h-8`, `h-6`, `py-1`). These need padding adjustments to ensure the clickable area is at least 44x44px even if the visual element appears smaller. Common offenders:
  - Close buttons on modals/drawers
  - Icon-only buttons
  - Small text links in the footer
  - Filter pills/tags on the events page
- For any icon-only buttons found, ensure they have `min-h-[44px] min-w-[44px]` or equivalent padding. Use padding to increase target area without changing visual size.

**5. 320px reflow verification prep (A11Y-07):**
- Search `src/app/globals.css` and any component files for any fixed widths (`w-[XXXpx]` where XXX > 300, `min-w-[XXXpx]` where XXX > 300, or `width: XXXpx` in inline styles) that could cause horizontal scroll at 320px viewport.
- Ensure no component uses `overflow-x: hidden` on the body or html -- this masks reflow issues rather than fixing them. If found on a wrapper element for a specific purpose (e.g., preventing horizontal scroll from an animation), that is acceptable.
- Check that the root layout does NOT have `<meta name="viewport" content="..., maximum-scale=1">` or similar viewport restrictions. The viewport tag should be the default Next.js one: `<meta name="viewport" content="width=device-width, initial-scale=1">` with no maximum-scale or user-scalable=no restrictions.

**A11Y-03: Semantic landmarks:**
Check that the layout structure uses correct semantic elements:
- `src/components/layout/Header.tsx` -- must use `<header>` element containing `<nav aria-label="Main navigation">`
- `src/app/layout.tsx` or each page -- the main content area must be wrapped in `<main id="main-content">`. If pages use `<main>` directly, that is fine. If there is no `<main>` element, add one. The `id="main-content"` is needed for the skip link.
- `src/components/layout/Footer.tsx` -- must use `<footer>` element
- If any page wraps content in a bare `<div>` instead of `<main>`, change it to `<main id="main-content">`
- Verify there is exactly ONE `<main>` element per page (not nested, not duplicated)

**A11Y-04: Strict heading hierarchy:**
Audit every page for heading level skips. Rules:
- Each page MUST start with exactly one `<h1>` (the page title)
- After h1, next heading must be h2 (not h3 or h4)
- After h2, next can be h2 (sibling) or h3 (child), but NOT h4
- After h3, next can be h2, h3, or h4, but NOT h5
- No heading level can be skipped (h1 -> h3 is invalid, h2 -> h4 is invalid)
- Common violations to look for:
  - Components that always render `<h3>` regardless of context (e.g., a card title that is h3 but used where h2 is expected)
  - Multiple h1 elements on a page (check for components that render their own h1)
  - Section headings using h3 when they should be h2

For each page, mentally trace the heading hierarchy and fix any skips. If a component renders a heading at a fixed level that causes a skip in certain page contexts, consider making the heading level a prop:
```tsx
type HeadingLevel = "h2" | "h3" | "h4";
function SectionTitle({ as: Tag = "h2", children }: { as?: HeadingLevel; children: React.ReactNode }) {
  return <Tag className="...">{children}</Tag>;
}
```

**A11Y-05: Image alt text:**
Search all `.tsx` files for `<img`, `<Image` (next/image), and any SVG usage:
- Meaningful images MUST have descriptive alt text: `alt="Farmer walking through a field at sunset"`
- Decorative images (backgrounds, accent graphics, dividers) MUST have `alt=""` (empty string, NOT missing alt attribute)
- SVG icons used inside buttons/links should have `aria-hidden="true"` on the SVG and the parent element should have `aria-label`
- Check lucide-react icon usage: icons inside buttons need `aria-hidden="true"` on the icon and `aria-label` on the button
- Common violations:
  - `alt={item.name}` where name is not descriptive (e.g., alt="John" -- should be "Photo of John Smith, Executive Director")
  - Missing alt attribute entirely
  - Alt text that describes the image file ("hero-image.jpg") instead of its content
