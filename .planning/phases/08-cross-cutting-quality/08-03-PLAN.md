**1. Create LazyMotion provider (PERF-03):**

Motion (Framer Motion) ships ~60KB of animation features by default. Using `LazyMotion` with the `domAnimation` feature set tree-shakes this to ~15KB, saving ~45KB.

Create `src/components/shared/LazyMotionProvider.tsx`:

```tsx
"use client";

import { LazyMotion, domAnimation } from "motion/react";

export function LazyMotionProvider({ children }: { children: React.ReactNode }) {
  return <LazyMotion features={domAnimation}>{children}</LazyMotion>;
}
```

IMPORTANT: Check which import path Motion uses in this project. It may be:
- `motion/react` (newer Motion v12+)
- `framer-motion` (older versions)

Check `package.json` for the package name and version. Use the correct import. If the package is `framer-motion`, import from `framer-motion`. If it is `motion`, import from `motion/react`.

**2. Wire LazyMotion into root layout:**

Open `src/app/layout.tsx` and wrap the children with the LazyMotionProvider:

```tsx
import { LazyMotionProvider } from "@/components/shared/LazyMotionProvider";

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en-GB" className={`${cormorantGaramond.variable} ${dmSans.variable}`}>
      <body>
        {/* JsonLd from Plan 02 */}
        <LazyMotionProvider>
          {/* SkipLink, UtilityBar, Header, etc. */}
          {children}
          {/* Footer */}
        </LazyMotionProvider>
      </body>
    </html>
  );
}
```

Preserve all existing layout content (SkipLink, UtilityBar, Header, Footer, JsonLd). Just wrap the animated content area with LazyMotionProvider.

**3. Update Motion components to use `m` instead of `motion`:**

When using LazyMotion, components should use the `m` component instead of `motion` for optimal tree-shaking. Search for all files that import from `motion/react` or `framer-motion`:

```bash
grep -rn "from.*motion" src/ --include='*.tsx'
```

For each file found:
- If it uses `motion.div`, `motion.section`, etc., change to `m.div`, `m.section`, etc.
- Update the import: `import { m } from "motion/react"` instead of `import { motion } from "motion/react"`
- Keep `AnimatePresence`, `useReducedMotion`, and other utilities as-is (they are not affected)

Common files to check:
- `src/app/template.tsx` (page transitions)
- `src/components/shared/AnimatedSection.tsx` (section reveals)
- Any component using `whileInView`, `animate`, `initial` props

If there are many components using `motion.*`, prioritize the most commonly used ones. At minimum, update:
- `template.tsx` (runs on every page transition)
- `AnimatedSection.tsx` (used on most pages)

**4. Image optimization audit (PERF-02):**

Search all `.tsx` files for image usage:

```bash
grep -rn "<img\|<Image\|src=.*\.\(jpg\|jpeg\|png\|webp\|avif\|svg\)" src/ --include='*.tsx'
```

For EVERY image found:
- It MUST use `next/image` (`import Image from "next/image"`), NOT raw `<img>` tags
- `next/image` automatically handles:
  - WebP/AVIF format conversion
  - Responsive srcset generation
  - Lazy loading by default (below the fold)
  - Width/height for CLS prevention
- If any raw `<img>` tags are found, convert them to `<Image>`:
  ```tsx
  // Before (bad):
  <img src="/images/hero.jpg" alt="..." />

  // After (good):
  <Image src="/images/hero.jpg" alt="..." width={1200} height={600} />
  ```
- For hero/above-the-fold images, add `priority` prop to disable lazy loading:
  ```tsx
  <Image src="/images/hero.jpg" alt="..." width={1200} height={600} priority />
  ```
- For decorative background images that are CSS backgrounds (not `<img>` tags), these are acceptable as-is but note them for potential optimization.

**5. Bundle size verification (PERF-01, PERF-04):**

After all changes, run the production build and analyze output:

```bash
npm run build
```

Next.js outputs a build summary showing route sizes. Check:
- **First Load JS** for each route must be under 100KB (the "First Load JS" column in build output includes shared chunks)
- If any route exceeds 100KB, investigate what is pulling in large dependencies
- Common culprits: Motion/Framer Motion (should be fixed by LazyMotion), date libraries, large icon imports

To get more detailed analysis, install and run the bundle analyzer:
```bash
npm install -D @next/bundle-analyzer
```

Create or update `next.config.ts` (or `.mjs`) to conditionally wrap with analyzer:
```ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  // existing config
};

// Only wrap with analyzer when ANALYZE=true
const withBundleAnalyzer = process.env.ANALYZE === "true"
  ? require("@next/bundle-analyzer")({ enabled: true })
  : (config: NextConfig) => config;

export default withBundleAnalyzer(nextConfig);
```

Then run: `ANALYZE=true npm run build`

This opens a browser page showing the bundle composition. Look for:
- Motion/Framer Motion should be ~15KB (not ~60KB) thanks to LazyMotion
- No duplicate React instances
- No large unused dependencies
- Total shared JS should be well under 100KB gzipped

If the bundle is over 100KB gzipped, investigate and fix by:
- Checking for barrel imports (`import { X } from "lucide-react"` should be `import X from "lucide-react/dist/esm/icons/x"` if tree-shaking is failing)
- Ensuring dynamic imports for heavy components
- Removing any accidentally imported server-side libraries in client components

**6. Page weight verification (PERF-01):**


Then use curl to check page size:
```bash
# Check total HTML size of homepage

# Or more accurately, check the full page weight including assets
# by looking at the build output sizes
```

The Next.js build output shows route sizes. Each page's "Size" column should be well under 500KB. This column shows the HTML + page-specific JS. Combined with shared JS (shown in the "First Load JS shared" line), total must stay under 500KB.

If any page is over 500KB:
- Check for large inline data (events list, team members with long bios)
- Check for large images not using next/image
- Consider pagination or lazy loading for data-heavy pages
